# CtoFConverter.com — 全新站点策略

> 本文档定义 ctofconverter.com 从「程序化 SEO 站点」转型为「最佳温度工具 + 知识中心」的完整方案。
> 核心目标：成为 Google 眼中值得收录的、真正对用户有价值的温度资源站。

---

## 一、核心理念

### 旧站已死的问题根源

| 旧站做法 | Google 判定 | 后果 |
|---|---|---|
| 文本纺写（textSpinner.ts） | 操纵搜索引擎，违反 Spam Policy | 降权、不收录 |
| 50 页模板 × 10 语言 = 500+ 近似页面 | 规模化内容滥用（Scaled Content Abuse） | 全部不收录 |
| 同一天 lastmod | 程序化生成信号 | 不被信任 |
| 广告优先于内容 | 用户价值低 | 低排名 |

### 新站黄金法则

1. **每页一个人写的** — 没有模板，没有 spin，每页有独特的信息增益
2. **工具做到最好** — 如果只是"输入数字点转换"，用户不需要你，Google 也不会推荐你
3. **用户搜什么，你写什么** — 不是"我有 50 个温度数字所以生成 50 页"，而是"用户在搜烤箱温度换算、室温多少合适"
4. **先有价值，后放广告** — 广告不嵌入内容流，放在不影响阅读的位置

## 三、GEO 策略（面向 AI 搜索优化）

### 什么是 GEO

GEO (Generative Engine Optimization) 是让内容被 AI 搜索引擎（ChatGPT、Perplexity、Google AI Overviews、Gemini、Claude）引用和推荐的优化方法。
与 SEO 的核心区别：
- **SEO**：排名 Google 前 10 名蓝色链接
- **GEO**：被 AI 直接引用在回答中

### GEO 核心做法

| 做法 | 说明 |
|------|------|
| **Answer Capsule** | 每个 H2/H3 下放 40-60 字的直接答案段落，方便 AI 提取引用 |
| **结构化数据** | 表格、列表、FAQ Schema、HowTo Schema |
| **实体优先写作** | 在文章开头明确命名实体（品牌、温度单位、场景），保持名称一致 |
| **可引用数据** | 引用权威来源（WHO、USDA、NOAA），提供精确数字 |
| **问答格式** | FAQ 块用 Schema 标记，AI 搜索优先引用 |
| **深度内容** | 3000+ 字，比竞品更全面 |
| **时效性** | 标注最后更新日期，定期刷新 |

### 每页 GEO 内容结构模板

```
【Answer Capsule】页面顶部 40-60 字，直接回答用户问题
├── 交互工具（核心差异化）
├── 核心内容（H2 分段，每段以 Answer Capsule 开头）
│   ├── 表格（结构化数据，AI 易解析）
│   ├── 列表（步骤、建议）
│   └── 引用（权威来源）
├── FAQ Schema 区块（每个问答 40-60 字）
└── 内部链接（连接其他相关页面）
```

### 内容质量标准

- **信息增益检查**：写完后问自己——用户在这个页面能得到什么在其他站看不到的？
- **不要写废话**：不要写 "Temperature is a measure of how hot or cold something is" — 没人搜这个
- **真实场景驱动**：不要写 "20°C = 68°F"，写 "20°C (68°F) is the ideal spring day — light jacket weather"
- **数据来源可查**：烤箱温度参考 USDA/WHO 等权威来源

### 严禁做法

- ❌ 文本纺写（同义替换）
- ❌ 相同结构换数字
- ❌ 过量广告（每页不超过 2 个广告位）
- ❌ 关键词堆砌
- ❌ 内容农场式的"大而全"但言之无物

## 五、差异化：工具要做的事

市场上温度转换器几百个。区分度在以下方向：

### 1. 智能上下文感知

用户输入一个温度，页面自动推断场景并显示相关内容：
- 35-42°C → 显示体温对照、发烧判断标准
- 150-220°C → 显示烤箱档位、烘焙建议
- -10-30°C → 显示穿衣建议、天气感觉
- 0°C / 100°C → 显示水的相变知识

### 2. 批量转换

```
输入:  0, 25, 37, 100, 180, 220
输出:  0°C = 32°F | 25°C = 77°F | 37°C = 98.6°F ...
       一键复制 / CSV 导出
```

### 3. 多单位同时显示

°C ↔ °F ↔ K (Kelvin) ↔ °R (Rankine)，在同一界面切换

### 4. 可分享的转换结果页

生成唯一 URL：`/result/37c-to-f` → 显示 37°C = 98.6°F 的完整结果页（可嵌入 OG）

### 5. 免费开发者 API（获取外链）

```
GET https://ctofconverter.com/api/convert?value=37&from=c&to=f
→ { "celsius": 37, "fahrenheit": 98.6, "formula": "(37 × 9/5) + 32" }
```

## 七、技术实施

### URL 规范

- **新页面全部不带尾部斜杠**：如 `/celsius-to-fahrenheit`、`/oven-temperature-conversion`
- **旧页面保持现有 URL**：无 `trailingSlash` 配置（Next.js 默认无斜杠），新旧 URL 一致
- 新旧并存方案：无需 middleware，直接在 `pages/` 下新建文件

### 与旧站关系

- 旧页面完全保留，不删除任何文件
- 新页面使用新的 URL 路径，不与旧页面冲突（撞名则直接替换旧文件内容）
- 旧页面逐渐不链接、不出现在 sitemap 中
- 新站站稳后，逐步将旧页面 301 到新站对应页面

### 技术栈

- Next.js (Pages Router)，React 19
- 新组件放在 `components/` 下新建文件
- 新页面放在 `pages/` 下，与旧页面同级
- 不需要修改任何旧文件

### 实施顺序

```
Phase 1 (现在 — 验证 Google 收录):
  ├── /oven-temperature-conversion        烤箱温度指南 [核心]
  │     策略：从 Trends 数据最强的方向切入，验证收录+排名
  │     竞品：Smeg(电商)、AllRecipes(食谱站)、CoolConversion(纯工具)
  │     差异化：四对照表 + 交互式换算工具
  └── /oven-to-air-fryer                 气炸锅转换 [子页面]
        策略：Trends 搜索量 100，上升趋势明显

Phase 2 (收录稳定后):
  ├── /room-temperature-guide             室温指南 [S 级量]
  ├── /conversion-formula                 公式+心算指南 [中量]
  └── 首页改造：添加新指南页导航区域

Phase 3 (权威提升后):
  ├── /cooking-temperature-chart          烹饪温度 [中量]
  ├── /weather-temperature-guide          天气穿衣 [低量，低竞争]
  └── /heat-index-chart / /wind-chill-chart

Phase 4 (完备):
  ├── /celsius-to-fahrenheit              主工具页 [最高竞争]
  ├── /batch-converter                    批量转换 [差异化武器]
  └── /temp-api                           免费 API [外链价值]
```

### 核心原则

- **每页完成后必须自检**：这个页面的信息增益在哪里？用户凭什么选择你而不是其他的页面？
- **不做就比做更好的判断**：如果某个主题你写不出比前 3 名更好的内容，就不做
- **先做 2 页验证**：不一口气做完，先看 Google 对 Phase 1 页面的收录和排名反应，再决定是否继续
